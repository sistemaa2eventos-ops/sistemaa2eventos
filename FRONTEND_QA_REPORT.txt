=========================================================
      RELATÓRIO DE AUDITORIA DE QA E FRONTEND
                   PROJETO: A2 EVENTOS
=========================================================

Como QA Automation e Desenvolvedor Frontend Sr, identifiquei oportunidades críticas para evitar "telas em branco" e duplicidade de dados em condições de estresse (uso rápido por operadores).

---------------------------------------------------------
🔴 1. RACE CONDITIONS: CLIQUES DUPLICADOS
---------------------------------------------------------
VULNERABILIDADE: O componente `NeonButton` já possui proteção (`disabled={loading}`), porém páginas vitais como `Pessoas.jsx` não gerenciam o estado de `saving`. 
CENÁRIO: Um operador impaciente clica 3 vezes em "Sincronizar Dados". A API recebe 3 requisições POST, gerando logs ou registros duplicados se o backend não for idempotente.

CORREÇÃO: Implementar `setSaving(true)` em todos os handlers de salvamento e vinculá-lo ao `loading` do `NeonButton`.

---------------------------------------------------------
🟠 2. PWA E MODO OFFLINE: REFRESH DE DADOS
---------------------------------------------------------
VULNERABILIDADE: O `LocalCheckinService` sincroniza os dados quando a rede volta, mas a UI não "percebe" que deve recarregar a lista de check-ins recentes ou as estatísticas do painel sem um F5.

CORREÇÃO: Emitir um Cross-Tab Event ou usar o `WebSocketService` para disparar um `sync_complete` que force o recarregamento dos componentes afetados.

---------------------------------------------------------
🟡 3. TRATAMENTOS DE ERRO E FRONTEND "BREAKING"
---------------------------------------------------------
VULNERABILIDADE: Falta um `ErrorBoundary` global. Se um componente falhar no render (ex: dado nulo inesperado), a aplicação inteira fica em branco. Além disso, erros 403 (Bloqueado) e 500 (Banco Offline) são mostrados via `alert()`, o que quebra a estética "Premium".

CORREÇÃO: 
- Envolver o `<App />` num `ErrorBoundary` customizado.
- Criar um interceptor no `api.js` que use o `Snackbar` do `notistack` para erros globais.

---------------------------------------------------------
🟢 PONTOS POSITIVOS
- PWA bem configurado via Vite (auto-update).
- Uso consistente de `notistack` para redundância básica.
- `App.jsx` já monitora eventos `online/offline`.
